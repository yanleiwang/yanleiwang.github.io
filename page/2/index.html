列表初始化也可以用在函数的返回值上

列表初始化的一些规则：

首先说下聚合类型可以进行直接列表初始化，这里需要了解什么是聚合类型：

    • 没有私有或保护的非静态数据成员
    • 没有基类
    • 没有虚函数

列表初始化的好处

个人认为列表初始化的好处如下：

  1. 方便，且基本上可以替代括号初始化
  2. 可以使用初始化列表接受任意长度
  3. 可以防止类型窄化，避免精度丢失的隐式类型转换

什么是类型窄化，列表初始化通过禁止下列转换，对隐式转化加以限制：

  • 从浮点类型到整数类型的转换
  • 从 long double 到 double 或 float 的转换，以及从 double 到 float 的转换，除非源是常量表达式且不发生溢出
  • 从整数类型到浮点类型的转换，除非源是其值能完全存储于目标类型的常量表达式
  • 从整数或无作用域枚举类型到不能表示原类型所有值的整数类型的转换，除非源是其值能完全存储于目标类型的常量表达式

类的访问权限(public，protected，private)

发表于 2022-03-17 更新于 2022-06-15 分类于 C/C++基础

首先明确访问修饰符和派生访问说明符指什么？

  • 类的成员函数可以直接访问作为其参数的同类型对象的私有成员

    • 意思就是说假如一个成员函数，有一个参数A是这个类类型，我能在该成员函数里，访问A的所有成员（public，protected和private）。

  • 一个类的友元函数就相当于这个类的成员函数。

    • 即，友元函数中可以访问类的所有成员。
    • 友元函数包括：
      • 设为友元的普通的非成员函数
      • 设为友元的其他类的成员函数
      • 设为友元类中的所有成员函数。

  • 类的对象只能访问类的public成员

  • 派生类的成员函数和友元函数可以访问基类的public, protected成员，不能访问private成员

    • 派生类的成员或友元只能通过派生类对象来访问基类的protected成员。派生类无法访问一个基类对象中的protected成员， 即

  • 派生访问说明符是控制派生类的用户（派生类的对象和派生的派生） 对于派生类中基类成员的访问权限

    • 也就是派生类中的基类成员是派生类的private还是protected或者public 成员
    • 基类的private成员永远是派生类的private成员，不管是什么继承
    • protected继承和private继承能降低基类的public和protected成员访问权限
    • 更具体的：
    • 使用public继承，基类的public、protected成员在派生类仍是public、protected
    • 使用protected继承，基类的public成员在派生类中变为protected、protected成员仍为protected
    • 使用private继承，基类的public、protected成员在派生类中变为private

    总结

  • private成员可以通过以下方式被访问

    • 该类中的成员函数
    • 友元函数

  • protected成员可以通过以下方式被访问

    • 该类中的成员函数
    • 友元函数访问
    • 派生类的成员函数
    • 派生类的友元函数
    • 通过public和protected继承，派生类的成员函数

  • public成员可以通过以下方式被访问

    • 该类中的成员函数
    • 友元函数访问
    • 派生类的成员函数
    • 派生类的友元函数
    • 通过public和protected继承，派生类的成员函数
    • 通过public继承，派生类的对象
    • 该类的对象

7.3 执行期类型识别(RTTI)

发表于 2022-03-17 分类于 深度探索C++对象模型

RTTI

RTTI，指在执行期获取指针所指对象的动态类型

  • RTTI 只支持多态类，也就是说没有定义虚函数是的类是不能进行 RTTI的。
    • 是因为RTTI的实现是通过vptr来获取存储在虚函数表中的type_info* ，事实上为非多态类提供RTTI,也没有多大意义
  • 对指针进行dynamic_cast失败会返回0 ,而对引用的话，识别会抛出bad_cast exception
    • 原因在于指针可以被赋值为0，以表示 no object，但是引用不行。
  • typeid 可以返回const type_info&，用以获取类型信息。
    • 虽然第一点RTTI只支持多态类，但typeid和type_info同样可用于内建类型及所有非多态类。与多态类的差别在于，非多态类的type_info对象是静态取得(所以不能叫“执行期类型识别”)，而多态类的是在执行期获得。

6.2 new和delete运算符

发表于 2022-03-15 更新于 2022-03-16 分类于 深度探索C++对象模型

new和delete语意

当我们使用一条new 语句时，实际执行了以下操作：

  1. 调用名为operator new的标准库函数，分配所需的内存
  2. 将配置得来的对象设立初值
    • 假如是内置对象，有显式初值，则设为显式初值，否则，其值是未定义的（上一次这块内存被使用时，遗留下来的值）
    • 假如是类对象，则调用对应的构造函数

delete语句类似：

  1. 调用相应的析构函数
  2. 调用名为operator delete的标准库函数，释放所需的内存

delete ptr之后的事情

当我们delete ptr之后，ptr所指的那块内存被释放，但是内存释放的时候，只是把这块内存标记为未使用，并没有清零（出于性能的考虑）， 也就是说delete ptr的操作，并没有去改变ptr指向的那块内存的值，并且ptr 的值是不变的，只是变成了野指针。但是我们无法保证那块内存会不会被程序其他地方再使用，里面的内容会不会被变更， 也就是说那块内存的内容是未定义的。

这也解释了以下问题：

  • 能否两次delete ptr？
    • 能否delete两次，取决于编译器，有的编译器比较严格，假如delete一块标记为未使用的内存会报错，而有的不会。但是考虑第二次delete ptr的时候，ptr所指的内存可能已经被程序别的地方使用了，此时delete就可能会导致错误，所以我们不应该两次delete ptr。
  • 使用一个野指针的问题？
    • 因为野指针所指的内存内容是未定义的，所以使用野指针的行为也是未定义的，可能会导致错误。

new[] 和 delete[]

当我们使用一条new[] 语句时，实际执行了以下操作：

  1. 调用名为operator new[]的标准库函数，分配所需的内存
  2. 挨个将配置得来的对象设立初值
    • 初始化方式进上面的代码

delete[]语句类似：

  1. 反方向，即从最后到最前面，挨个调用相应的析构函数
  2. 调用名为operator delete[]的标准库函数，释放所需的内存

delete[] 详解

  • delete[]如何知道数组的维度？
    • 在new[]语句分配的内存，额外分配一块区域用来存储数组的维度。
  • delete 和 delete[] 语句的区别？
    • delete[]语句时，编译器才会知道要寻找数组的维度，否则它便假设只有单独一个objects要被删除，只会调用一次析构函数，所以假如对于本该使用delete[]，而错误的使用了delete语句，当对象的析构函是有用的时候（比如释放堆上的内存），就会导致错误。

6.1 对象的构造和析构

发表于 2022-03-15 分类于 深度探索C++对象模型

前言

当程序离开当前作用域的时候，会调用当前作用域所有对象的析构函数，因此为了减少不必要的构造对象和析构对象的代码被插入到自己的代码当中，尽可能将对象定义在接近要使用的地方。

全局对象

一个全局对象，c++保证它在main()在第一次使用它之前将其构造，而在main()结束之前，将之析构掉。全局对象的初始化顺序如下：

  • 编译期，将全局对象分配至global区内存空间
    • 假如使用常量表达式来进行初始化全局对象，则设置值
    • 否则，由于global区内存空间全是0比特，全局对象的值为0
  • 运行期，在main函数的一开始，调用对应构造函数初始化所有全局对象。

缺点

使用全局对象的坏处是：

  • 不同文件中的全局对象构造顺序未定义，如果互相之间存在依赖关系的话，会产生未知的结果
  • 一个全局对象如果构造过程中抛出异常，无法来捕获并处理这个异常，所以会导致程序的终结

所以书中建议：根本就不要使用那些类类型的全局对象。如果真的需要一个全局对象，那么用一个函数封装一个局部静态对象，也是一样的效果。

局部静态对象

局部静态对象的写法：

C++标准规定，局部静态对象在第一次被调用时被构造，而在整个程序退出之时按构造相反的顺序析构局部静态对象。

对于上述机制，有一种实现策略：

  • 添加一个全局变量（类型为指向该局部静态变量的指针，初始值为nullptr）用来标识有没有初始化该局部静态变量，当第一次调用函数，碰到处局部静态变量，全局变量值为nullptr，于是调用构造函数，随后将全局变量值设置为该局部静态变量的地址。而当程序退出的时候，根据这个全局变量，决定是否调用相应析构函数。

对象数组

对于定义一个普通的数组，例如：

Point knots[ 10 ];

实际上背后做的工作则是：

  • 分配充足的内存以存储10个Point元素
  • 为每个Point元素调用它们的相应的构造函数
    • 如果显示提供了构造函数，则调用对应的构造函数，不足的调用默认构造函数(如果有的话，且不论是合成的还是显式定义的)，比如
      • point i[10] = {point(), point(1,2,3)}）

当数组的生命周期结束的时候，则要逐一调用析构函数，然后回收内存。

5.4 析构语意学

发表于 2022-03-15 分类于 深度探索C++对象模型

前言

如果类没有定义析构函数，那么只有在类内置的数据成员或者类的基类拥有程序员定义的析构函数的情况下，编译器才会合成出一个。否则，析构函数将会被视为不需要，自然也就不需要被合成（更不需要被调用）。

什么时候需要程序员定义析构函数

书中提到一个值得注意的问题, 并不是定义了构造函数就需要定义析构函数, 这种”对称”是无意义的. 只有当需要一个析构函数时, 我们才应该显式定义之. 那么什么时候需要呢? 首先要搞清楚析构函数的作用, 她是对象的生命周期的终结, 而函数体内执行的主要是是对对象持有的资源的释放, 例如在构造函数中动态申请的空间.

析构函数操作顺序

析构函数的操作与构造函数的顺序相反：

  • 本身的析构函数被执行。
  • 以声明的相反顺序调用member object 的析构函数
  • 重设vptr 指向适当的基类的虚函数表
  • 以声明相反的顺序调用基类的析构函数
  • 如果当前类是 most-derived class（最底层类），那么以构造的相反顺序调用虚基类的析构函数。

5.3 拷贝赋值运算符语意学

发表于 2022-03-13 更新于 2022-03-15 分类于 深度探索C++对象模型

前言

当我们设计了一个class，并以一个class object指定给另一个class object时，我们知道会调用拷贝赋值运算符，而对于这个拷贝赋值运算符我们有三种选择：

  1. 让编译器隐式生成一个默认的拷贝赋值运算符，方法有
    • 把拷贝赋值运算符声明为=default (C++11 推荐做法)
      • 类名& operator=(const 类名&)=default
    • 不定义拷贝赋值运算符
  2. 程序员定义的显式拷贝赋值运算符
  3. 明确拒绝复制操作，方法有：
    • 把拷贝赋值运算符声明为=delete (C++11 推荐做法)
      • 类名& operator=(const 类名&)=delete
    • 将*operator =*声明为private, 并且不提供其定义 (C++ 11 之前的做法)

编译器版本拷贝构造运算符语意

编译器版的拷贝构造运算符和编译器版的拷贝构造函数语意类似，其行为是memberwise copy，即挨个拷贝数据成员。

当memberwise copy表现出bitwise copy（挨个字节拷贝）的语意的时候，认为其是trivial （无用的），根本不会合成出来，当满足下面的条件时，就是trivial的：

  • 类没有虚函数.
  • 类没有虚基类.
  • 直接基类的复制赋值运算符都是trivial的.
  • 非静态成员的复制赋值运算符是trivial的.

虚继承的拷贝抑制

不管是程序员自定义的，还是编译器合成的拷贝赋值运算符，都需要考虑一个问题：在如下继承体系下，当拷贝Vertex3d对象的时候，如何避免虚基类Point的拷贝赋值运算符多次调用。

与5.2 构造函数语意学里说到的在虚拟继承情况下虚基类被拷贝多次是一个意思，然而拷贝赋值运算符其实并没有特别好的办法避免多次拷贝虚基类。

不过幸运的是：编译器合成版本和程序员自定义的拷贝赋值运算符并不会发生错误，拷贝赋值运算符不像构造函数一样，会被编译器默默加很多代码（比如调用基类的构造函数，设置vptr等等），所以虚基类被拷贝多次并不会带来错误，只有效率上的影响（构造函数需要避免多次构造虚基类的主要原因是因为：派生类会首先虚基类的构造函数，然后调用基类的构造函数，而基类的构造函数同样会调用虚基类的构造函数，如果这两次构造出来的虚基类不一致（比如传入不同的参数）就会发生错误）。

程序员定义的显式拷贝赋值运算符

需要考虑是不是显式提供一个operator =. 一个原则是:

​ 只有在默认复制赋值运算符的行为不安全或不正确时, 才需要显式定义复制赋值运算符.

由于拷贝赋值运算符不像构造函数一样，会被编译器默默加很多代码，所以多次调用虚基类的拷贝赋值运算符不会带来错误，但是为了确保虚基类被正确的拷贝，我们可以把虚基类的拷贝赋值操作放在最后，调用方式：类名::operator=(对象名)，当然书中建议：不允许任何一个virtual base class的拷贝操作，或者更进一步地，不在任何virtual base class中声明数据。

5.2 构造函数语意

发表于 2022-03-13 分类于 深度探索C++对象模型

真实的构造函数做了哪些事情

编译器会扩充每一个constructor，扩充程序视当前class T的继承体系而定，一般而言编译器所做的扩充操作大致如下：

  1. 调用所有虚基类的构造函数，从左到右，从最深到最浅：

    • 调用虚基类的构造函数的大前提是当前构造函数构造的类是最底层的（most-derived class），直观点说就是，虚基类的构造由最外层类控制。

    • 如果虚基类被列入初始化列表中，那么调用初始化列表中的相应构造函数，否则调用默认构造函数（如果默认构造函数是delete的话，报错）

    • 此外，要保证虚基类的偏移量在执行期可存取

      • 对于使用指向虚基类的指针（vbptr）来实现虚基类的编译器来说就是对vbptr的初始化。
      • 如果是使用虚函数表保存偏移量那应该是推迟到虚表指针（vptr）的设定再进行（我猜的）

  2. 调用所有基类构造函数，依声明顺序：

    • 如果该基类被列入了成员初始化队列，那么调用初始化列表中的相应构造函数，否则调用默认构造函数（如果默认构造函数是delete的话，报错）
    • 如果该基类是第二顺位或之后的基类，那么在调用该基类的构造函数时，this 指针参数必须被调整才能传递过去，需要是派生类地址+偏移。

  3. 正确初始化vptr

  4. 按照非静态数据成员的声明顺序，初始化所有非静态数据成员

    • 如果数据成员是内置类型
      • 如果被列入成员初始化队列，那么根据初始化列表中的初始值进行初始化
      • 否则，实际不初始化，其值未知
    • 如果数据成员是类类型
      • 如果被列入成员初始化队列，那么调用初始化列表中的相应构造函数
      • 否则调用默认构造函数，如果默认构造函数是delete的话，报错

  5. 执行构造函数体

虚继承的构造抑制

调用虚基类的构造函数的大前提是当前构造函数构造的类是最底层的（most-derived class）。其目的是为了保证这个虚基类子对象只初始化了一次。

否则考虑如下继承体系，

当我们定义Vertex3d cv;时，Vertex3d的构造函数中会调用Point的构造函数、而随之调用它的子对象，Point3d和Vertex的构造函数中也调用了Point的构造函数。

先不说，对于同一个子对象进行三次初始化是否有效率，更重要的是，这将不可避免的带来错误。由Vertex3d指定的子对象Point的值，会被覆盖掉（Point3d和Vertex各自的构造函数可能调用Point不同的构造函数，或者传了不同的实参）

根据c++ 语法，Point 的初始化应有most-derived class来施行。也就是说当Vertex3d为最底层的时候，应当由它的构造函数来调用Point的构造函数初始化Point子对象，Vertex3d的子对象的构造函数对于Point的调用则应当抑制。

如何抑制

有两种方法：

  • 给所有构造函数增加一个参数，一个bool值，用来表示是否为most-derived class，也就是判断当前构造函数所在类是否是继承体系的最底层. 各构造函数根据这个条件变量来决定是否调用虚基类的构造函数，因此通过控制这个条件变量，就可以抑制非most-derived class调用虚基类的构造函数。

  • 所以另外一种方法，是提供两种构造函数, 一种针对完整的object，一种针对subobject，完整的无脑上虚基类构造函数，并且调用基类的subobject版本构造函数，不执行虚基类构造函数。
