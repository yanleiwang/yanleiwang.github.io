    • 或许你可以通过其派生类来初始化它的数据成员（假如nostatic data member为public或protected）,但这样做的后果则是破坏了数据的封装性，使类的维护和修改更加困难。由此引申，类的data member应当被初始化，且只在其构造函数或其member function（供构造函数调用）中初始化。

    • 必须定义纯虚析构函数，而不能仅仅声明它的原因在于：每一个继承类的析构函数会被编译器加以扩展，以静态调用方式其每一个基类的析构函数（假如有的话，不论是显示的还是编译器合成的），所以只要任何一个基类的析构函数缺乏定义，就会导致链接失败。所以即使纯虚函数在语法上允许我们只声明而不定义纯虚函数，但还是必须实现该纯虚析构函数。
    • 因此假如有其他纯虚函数，尽量不要将析构函数设计为纯虚的，除非找不到其他纯虚函数。

  • 真的必要的时候才使用虚函数，不要滥用虚函数。虚函数意味着不小的成本：

可变参数模板

  • 可变参数模板(variadic template)为一个接受可变数目参数的模板函数或模板类。
  • 参数包(parameter packet)可变数目的参数。
  • 模板参数包(template parameter packet)表示零个或多个模板参数。
  • 函数参数包(function parameter packet)表示零个或多个函数参数。

用省略号指出一个模板参数或函数参数表示一个包。

C++存储说明符和CV限定符（static，extern，thread_local，mutable，const，volatile）

发表于 2022-03-10 分类于 C/C++基础

存储持续性

C++使用四种不同的方案来存储数据，这些方案的区别在于数据的存储持续性和存储位置。

  • 自动存储持续性。局部变量的存储持续性为自动，存储在函数栈中，在程序执行所属函数时被创建，执行完时所用内存被释放。
  • 静态存储性。全局变量和使用static定义的变量的存储持续性都为静态，存储区域为全局数据区（细分为静态存储区、data区和bss区），在程序整个运行过程中都存在。
  • 线程存储持续性。使用关键字thread_local声明的变量，其生命周期与所属的线程一样长。
  • 动态存储持续性。存储在堆中的内存，除非显式释放（delete），否则在程序整个运行过程中都存在。

链接性

链接性描述了名称（变量和函数）如何在不同的编译单元（源文件）间共享，C++一共有三种链接性：

  • 内部链接性，具有该链接性的名称只能在定义它的编译单元内可见
  • 外部链接性，具有该链接性的名称可以在所有文件间共享。
  • 无链接性，只能在当前函数或代码块中访问。（其实也就是局部变量具有该链接性）

普通变量和函数的存储持续性和链接性

  • 一个普通的全局变量，生命周期是全局的，整个程序的运行过程中都存在，存储区域为全局数据区，具有外部链接性。由于全局数据区的内存，在程序启动的时候，会被全部初始为0，假如变量没有显示初始化，如果是类类型，则会执行默认构造函数；如果是内置类型，其值为0.
  • 一个普通的全局函数，生命周期是全局的，具有外部链接性
  • 一个普通的局部变量，存储在函数栈中，在程序执行所属函数时被创建，执行完时所用内存被释放，无链接性。

变量和函数可以加存储说明符和CV限定符改变其存储持续性和链接性。

存储说明符

C++一共有五种存储说明符可以改变存储持续性和链接性（在同一个声明中不能使用多个说明符，thread_local除外，它可与static或extern结合使用）：

  • register
  • static
  • extern
  • thread_local
  • mutable

register

  • 只有局部变量才能用register修饰。

  • 在C和C++11之前，该关键字的作用是建议编译器使用CPU寄存器来存储该局部变量。而C++11中，该作用已经失去了，只是显式地指出该变量是局部变量，然而也只有局部变量才能用register关键字修饰，所以其实就是没用，之所以还保留，是为了避免使用了该关键字的现有代码非法。

static

static关键字可以修饰一个函数，一个变量，也可以修饰一个类成员

  • 当修饰一个全局变量或者一个全局函数的时候，修改的是这些符号的链接，从外部链接性变为内部链接性，这些符号只能在定义它的源文件中被使用，其他的源文件不能被使用，而存储位置和生命周期都不变。
    • 所以我们可以在两个源文件里定义两个同名的static全局变量/函数，而不会产生重定义的编译错误，这两个static变量的内存地址是不同的。
    • static全局变量/函数应该定义在源文件中。
  • 当修饰一个局部变量的时候，改变了变量的存储位置和生命周期，普通的局部变量存储位置是在栈中的，并且在函数调用完之后被自动释放，而static局部变量的存储位置是在静态存储区，假如未经初始化会被自动初始化为0。static局部变量在函数的第一次执行的时候被创建，程序结束的时候被销毁。它的作用域不变。
  • 假如修饰的是一个类成员，表示的是这个数据成员或者成员函数是属于类的，而不是某个对象的，所有对象共享一个静态成员，并且静态成员函数，由于没有this指针，所以不能访问类的非static成员。

模板实参推断

形参是T

我们传递这6种实参，实例化后的模板函数func，类型T都推断为int，因为此时是传值，并且顶层const会被忽略（所以传递const int ci，T也推断为int）

形参是T&

当函数参数是模板类型参数的一个普通左值引用时（即，T&），绑定规则告诉我们，只能传递一个左值，所以情况6）会编译错误，而其他情况正常（不要忘了，右值引用是一个左值），实参可以是const类型或非const类型，而根据底层const不会被忽略规则，假如实参是const，则T被推断为const。

当我们传递一个右值引用给模板函数时，类型T推断为int &&，而根据引用折叠规则，实际上实例化的函数为 void func(int& val), 也就是说假如我们在函数体内改变了val的值，会影响到外面的值。

形参是const T&

如果一个函数参数类型是const T&，正常绑定规则告诉我们可以给它传递任何类型实参

  • 一个对象
    • const
    • 非const
      • 因为可以将一个非const对象的引用（或指针）传递给一个const的引用（或指针）形参
  • 纯右值
    • 不要忘了，纯右值可以用来初始化一个常量左值引用

当函数参数本身是const时，T的类型推断的结果不会是一个const类型，因为const已经是函数参数类型的一部分了，它不会再是模板参数类型的一部分。

同样当我们传递一个右值引用给模板函数时，类型T推断为int &&，而根据引用折叠规则，实际上实例化的函数为 void func(const int& val)

形参是T&&

当一个函数参数是一个右值引用时，显然我们可以传递一个右值给它，所以情况6），毫无疑问，类型T被推断为int。

但是情况1-5，显然是一个左值，正常情况下，我们是无法把一个右值引用绑定到左值身上的，但是C++语言指定了一个例外：当将一个左值传给一个参数为右值引用的模板函数（T&&），此时编译器会推断模板类型参数T为实参的左值引用类型（即，T&），同时根据底层引用不会被忽略规则， 情况2和4，T为const int&

所以一个模板函数形参是右值引用的参数，可以给它传递任何类型实参

引用折叠

当形参是T&&， 传递给函数int&的时候，我们可以看到函数的形参val类型是int& &&，也就是引用的引用这种情况，这种情况下，这些引用会形成“折叠”，折叠成左值的引用或者右值的引用。

  • X& &、X& &&、X&& &都被折叠成X&
  • 类型X&& &&被折叠成X&&

比如当我们用一个左值调用func时，希望func函数体内调用的是左值版print，当用一个右值调用func时，希望右值版的print被调用。

值类别

自C++11开始，表达式的值分为

  • 左值(lvalue, left value)

  • 将亡值(xvalue, expiring value)

  • 纯右值(pvalue, pure ravlue)

  • 以及两种混合类别：

    • 泛左值(glvalue, generalized lvalue, 左值+将亡值)
    • 右值(rvalue, right value， 纯右值+将亡值)

左值

  • 左值是有实际存储地址的值（具名），可以通过取地址符&获取其地址
  • 常见的左值类型
    • 变量名、函数名以及数据成员名
    • 返回左值引用的函数调用
    • 字符串字面值（如”abc”）

将亡值

  • 将亡值，顾名思义是即将消亡的值，是有临时存放位置（具名），但是马上要销毁的值（可移动）。
  • 是跟右值引用相关的表达式，只能通过以下方式来获得：
    • 返回右值引用T&&的函数返回值
    • std::move的返回值
    • static_cast<T&&>(t);

纯右值

  • 没有存放位置（非具名），且不可移动的值
  • 常见的纯右值类型：
    • 字面值(字符串字面值除外)，例如1，’a’, true等
    • 返回值为非引用的函数的返回值
    • 算术表达式，比如 a+b
    • 逻辑表达式
    • 比较表达式

  • 字符串字面值是左值，其他字面值都是纯右值，一个很重要的原因，就是可以字符串字面值可以获取地址，这是因为C++将字符串字面值实现为char型数组，实实在在地为每个字符都分配了空间并且允许程序员对其进行操作。如果从存储区的概念来理解，那就是字符串字面值存储在常量区。

模板的实例化

一个模板只有被使用到，才会被实例化，否则不会被实例化。对于一个实例化后的模板来说，未被调用的成员函数将不会被实例化，只有成员函数被使用时，C++标准才要求实例化他们。其原因，有两点：

  • 空间和时间效率的考虑

  • 使模板有最大的适用性。

4.2 虚函数实现模型

发表于 2022-03-05 更新于 2022-03-12 分类于 深度探索C++对象模型

在C++中, 多态表示“以一个基类的指针或引用，寻址出一个子类的意思

虚函数的一般模型：

  • 每一个声明有（或继承自）一个或多个虚函数的类有一个虚函数表(一个类只有一个)，里面包含了这个类所有虚函数的地址和该类类型信息

  • 这个类的每个对象有一个虚表指针，指向这个虚函数表。

  • 给每一个虚函数指派一个在表中的索引。

单继承模型

实现机制

单继承虚函数的实现机制就是虚函数的一般模型，用这种模型来实现虚函数得益于在C++中,虚函数的地址在编译期是可知的，而且这一地址是固定不变的。而且表的大小不会在执行期增大或减小。

虚表中有什么？

一个类的虚函数表中存储类型信息（通常存储在索引为0的位置）和所有虚函数地址。这些虚函数包括三种：

  • 这个类定义的虚函数，假如基类也定义有这个虚函数，那会改写虚表中相同索引值存储的地址。
  • 继承自基类的虚函数实体，——基类定义有，而这个类却没有定义。直接继承之。
  • 一个纯虚函数实体。用来在虚函数表中占座，有时候也可以当做执行期异常处理函数。

每一个虚函数都被指派一个固定的索引值，这个索引值在整个继承体系中保持前后关联，例如,假如z()在基类虚函数表中的索引值为2，那么在子类的虚函数表的索引值也为2。

虚表的构建步骤

当一个类单继承自有虚函数的基类的时候，将按如下步骤构建虚函数表：

  • 继承基类中声明的虚函数——这些虚函数的实体地址被拷贝到继承类中的虚函数表中相同的slot中。
  • 如果有改写（override）基类的虚函数，那么在1中应将改写（override）的函数实体的地址放入对应的slot中而不是拷贝基类的。
  • 如果有定义新的虚函数，那么将虚函数表扩大一个slot以存放新的函数实体地址。

  • 带有多态性质的 base class 应该声明一个 virtual 的析构函数。

  • 析构函数绝对不要吐出异常。如果一个被析构函数调用的函数可能抛出异常，析构函数应该捕获任何异常，然后吞下它们（不传播）或者结束程序，并且用日志记录异常。

    • 因为只要析构函数吐出异常，程序就可能过早结束或出现内存泄漏

      • 因为当异常发生的时候，c++的机制会调用已经构造对象的析构函数来释放资源，比如我们为某个类A创建size为10的vector，挨个为它执行默认构造函数，假如执行到第5个元素的时候，抛出异常了，c++这时候又会从第4个到第1个倒着执行析构函数，假如这时候析构函数又抛出了异常，ok，那现在程序里面有两个未被解决的异常，那程序直接被结束掉。

      • 并且假如析构函抛出异常，那么函数体内之后的代码就不会被执行，而如果析构函数在异常点之后需要执行某些必要的动作比如释放某些资源，则这些动作不会执行，这就会造成诸如资源泄漏的问题。

  • 当我们写一个库的时候，如果客户（使用这个库的程序员）需要对某个操作函数运行期间抛出的异常做出反应，那么class应该提供一个普通函数（而非在析构函数中）执行该操作。

    • 这样选择权就在客户手里。如果客户认为该动作决不会抛出异常，就不必调用新函数；但若因为客户忘调用新函数，导致析构函数抛出异常进而导致程序进入不确定行为，锅也不在设计者头上，毕竟客户自己放弃了处理异常的机会。

条款 09：绝不在构造和析构过程中调用 virtual 函数

  • 在构造和析构期间不要调用virtual函数，因为这类调用从不下降至派生类
    • 当执行派生类的构造函数的时候，会首先执行基类的构造函数，然后执行派生类的构造函数，而在执行各自的构造函数的时候，执行函数体之前，会设置vptr指向各自的虚函数表。因此在执行基类的构造函数时候，vptr指向的是基类的虚函数表，所以基类构造函数中调用的虚函数，实际执行的是基类版本的，并没有多态的表现。
    • 析构函数，同理执行完派生类的析构函数函数体之后，会设置vptr指向基类的虚表。

条款 19：设计 class 犹如设计 type

在设计class之前，设计者必须扪心自问：

  • 新type的对象应该如何被创建和销毁？
    • 这决定了构造函数、析构函数、内存分配和释放函数如何设计
  • 对象的初始化和赋值有何区别？
    • 决定了构造函数与赋值操作符的的行为，以及差异。尤其需要注意的是，“初始化”、“赋值”是完全不同的概念，它们调用了不同的函数。
  • 新type的对象如果被pass by value，意味着什么？
    • 记住，拷贝构造函数定义了一个新类型的传值（pass-by-value）如何实现。
  • 新type的合法值是什么？
    • 通常，对于一个类的数据成员来说，仅有某些值的组合是合法的。那些组合决定了你的类必须维持的不变量。这些不变量决定了你必须在成员函数内部进行错误检查，特别是你的构造函数，赋值运算符，以及 “setter” 函数。它可能也会影响你的函数抛出的异常，以及你的函数的异常规范（exception specification）（你用到它的可能性很小）。
  • 新type需要配合某个继承体系吗？
    • 如果当前type继承自某些class，那你必然受到了那些class设计的束缚，尤其是virtual或者non-virtual成员函数的影响。
    • 如果你允许别的类继承该type，那请谨慎考虑析构函数是否需要virtual。
  • 新type需要什么样的转换？
    • 如果你允许T1被隐式转换为T2，就必须在T1内写一个类型转换函数，或者为T2写一个non-explicit-one-argument(可被单一实参调用）的构造函数。
    • 如果你只允许explicit构造函数存在，就必须撰写显式类型转换函数，并且禁止定义类型转换操作符或者non-explicit-one-argument构造函数。
  • 新type需要哪些操作符和成员函数？
    • 该问题的回答决定了class中需要声明的函数，其中有一些是成员函数，有些则不应该是。（Effective C++ 24,25,47）
  • 新type应当明确禁止哪些函数？
    • 该问题的回答决定了class中哪些函数需要声明为没有定义的private成员函数。（Effective C++ 6）
  • 谁会试图使用新type的成员？
    • 该问题的回答决定了成员函数的可见性，并且也决定了friend。
  • type的一般化程度如何？
    • 如果一般化的程度足够高，或许我们不应该定义一个class，而是应该定义一个class template。
  • 是否真的需要一个新type？
    • 如果你只是想增加了一个derived class为原有的class添加功能，先试试看能不能用一些non-member函数或者templates解决。

条款 20：宁以 pass-by-reference-to-const 替换 pass-by-value

  • 前者通常比较高效，并可避免切割问题
    • 传值方式传递对象，会使用copy构造函数，对于一些体量庞大的对象来说，pass-by-value 是非常耗时的操作
    • 除了效率方面的考量，传引用也可以避免对象切割的问题。当一个 derived class 对象以 by value 的方式传递到函数中，而函数参数的类型是 base class 类，那么 base class 的 copy 构造函数会被调用，导致传入的对象变成了 base class ，绝不会是你想要的 derived class。
  • 该规则并不适用内置类型，以及STL的迭代器和函数对象。对它们而言，pass-by-value往往比较适当
  • 引用在 C++ 编译器的底层中，往往通过指针实现。

条款 21：必须返回对象时，别妄想返回其 reference

  • 绝不要返回指针或引用指向一个局部栈对象
  • 返回引用指向一个堆对象
  • 返回指针或引用指向一个局部静态对象而有可能同时需要多个这样的对象。

条款 22：将成员变量声明为 private

类的成员变量最好是 private, ，然后使用函数做读/写访问。有两大原因：

  • 使用函数控制可以让成员变量的处理有更精密的控制，比如，你可以通过设置 getter 和 setter 实现只读访问、不准访问、读写访问，甚至只写访问的成员变量。

  • 封装。将成员变量隐藏在函数接口的背后，可以为所有可能的实现提供弹性

    • 如果有一个成员变量是 public ，并且在后续代码中，直接读写这个成员变量。那么，一旦你需要实现当该变量被写时通知其他对象或者控制该变量的正确赋值范围，你会发现很多代码都需要变更！这是一个非常可怕的工作量。

条款 23：宁以 non-member non-friend 替换 member 函数

non-member non-friend 函数： 普通函数(不是成员函数，且不是友元函数）

面向对象守则要求数据应该尽可能地被封装，而不是将数据以及操作数据的那些函数被捆绑在一块。

non-member non-friend 函数比member函数好在：

  • 更高的封装性
  • 只有 class 的成员函数和其 friend 函数能够直接操控private成员。而增加一个member 函数就意味着增加一个可以窥见类内部秘密的“知情人”，其封装性就会减弱。
  • 较低的编译相依度
    • 成为类的 non-member non-friend 函数并不意味它不可以是另一个 class 的 member ，比如可以设立某些工具类的 member 函数。在 C++ 中，更加自然的做法就是让 class 和这些 non-member non-friend 函数存在于同一个 namespace 中。因为 namespace 可以跨文件，那么这些 non-member non-friend 函数就可以按分类放在不同的文件内，当客户需要时，再 #include 相应的头文件，以此降低编译依赖性。C++ 的标准程序库就是这样组织起来的。

条款27：尽量少做转型动作

  • 转型语法：

    • const_cast：能够把常量对象转为非常量，也只有它拥有该能力
    • dynamic_cast：主要用来执行“安全向下转型”，也就是把基类指针转为派生类指针。它是唯一一个无法由旧式语句转换的新类型转换，也是唯一一个可能耗费大量运行成本的转型操作。
      • 绝对必须避免的是连锁式使用dynamic_cast，其运行效率极低且缺乏维护性。这种程序总应该被某种基于virtual函数调用的方法取而代之。
    • reinterpret_cast：把一个整型转为指针类型，反之亦然。不可移植，慎用（俺也不懂）
    • static_cast：强迫隐式转换，比如把non-const转为const，int转为double。它也可以作上述转换的反向转换，例如把派生类指针转为基类指针，或者把void*指针转为typed指针。但是没法把const转为non-const。

  • 如果可以，尽量避免转型，特别是在注重效率的代码中避免dynamic_casts。如果有个设计需要转型动作，试着发展无需转型的替代设计

  • 如果转型是必要的，试着将它隐藏于某个函数背后。客户可以调用该函数，而不需要将转型放进他们自己的代码内
