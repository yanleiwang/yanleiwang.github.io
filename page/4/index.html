条款 07：为多态基类声明 virtual 析构函数

  • 带有多态性质的 base class 应该声明一个 virtual 的析构函数。

    • 这是为了保证当我们delete一个基类指针， 而这个指针实际指向的是子类的时候，能够调用子类的析构函数，而不是基类的，防止发生内存泄漏。

  • 类的设计如果不是作为基类使用，或不是为了具备多态性，就不该声明virtual析构函数

    • 因为为了支持虚函数，会增加内存空间的开销（vptr）
    • 所以最佳实践是：如果类中包含虚函数，则该类的析构函数也应该是虚函数。

  • 当我们声明一个抽象基类（接口类），又想不到其他的 纯虚函数时，可以把析构函数定义为纯虚函数，然而我们必须在类外定义这个纯虚析构函数！

    • 这是因为，子类的析构函数会调用父类的析构函数，所以父类的析构函数必须定义！！

  • STL 中的标准容器类和string类的析构函数都是 non-virtual 的，因此不应该继承它们。

条款 08：别让异常逃离析构函数

  • 析构函数绝对不要吐出异常。如果一个被析构函数调用的函数可能抛出异常，析构函数应该捕获任何异常，然后吞下它们（不传播）或者结束程序，并且用日志记录异常。

    • 因为只要析构函数吐出异常，程序就可能过早结束或出现内存泄漏

      • 因为当异常发生的时候，c++的机制会调用已经构造对象的析构函数来释放资源，比如我们为某个类A创建size为10的vector，挨个为它执行默认构造函数，假如执行到第5个元素的时候，抛出异常了，c++这时候又会从第4个到第1个倒着执行析构函数，假如这时候析构函数又抛出了异常，ok，那现在程序里面有两个未被解决的异常，那程序直接被结束掉。

      • 并且假如析构函抛出异常，那么函数体内之后的代码就不会被执行，而如果析构函数在异常点之后需要执行某些必要的动作比如释放某些资源，则这些动作不会执行，这就会造成诸如资源泄漏的问题。

  • 当我们写一个库的时候，如果客户（使用这个库的程序员）需要对某个操作函数运行期间抛出的异常做出反应，那么class应该提供一个普通函数（而非在析构函数中）执行该操作。

    • 这样选择权就在客户手里。如果客户认为该动作决不会抛出异常，就不必调用新函数；但若因为客户忘调用新函数，导致析构函数抛出异常进而导致程序进入不确定行为，锅也不在设计者头上，毕竟客户自己放弃了处理异常的机会。

条款 09：绝不在构造和析构过程中调用 virtual 函数

  • 在构造和析构期间不要调用virtual函数，因为这类调用从不下降至派生类
    • 当执行派生类的构造函数的时候，会首先执行基类的构造函数，然后执行派生类的构造函数，而在执行各自的构造函数的时候，执行函数体之前，会设置vptr指向各自的虚函数表。因此在执行基类的构造函数时候，vptr指向的是基类的虚函数表，所以基类构造函数中调用的虚函数，实际执行的是基类版本的，并没有多态的表现。
    • 析构函数，同理执行完派生类的析构函数函数体之后，会设置vptr指向基类的虚表。

条款 11：在 operator= 中处理“自我赋值”

对于持有资源的类（比如有个数据成员是指针类型），自我赋值往往会发生问题。

这里的 self-assignment（自赋值）问题在 operator= 的内部，*this（赋值的目标）和 rhs 可能是同一个 object（对象）。如果它们是，则那个 delete 不仅会销毁 current object（当前对象）的 bitmap（位图），也会销毁 rhs 的 bitmap（位图）。

解决方法有3种：

  • 证同测试，即 在函数开头，插入代码if (this == &rhs) return *this;

    • 证同测试虽然检验了自我赋值的安全性，但不具备“异常安全性”，因为先执行delete pb，后执行 pb = new BitMap(*rhs.pb); 时可能产生异常，此时 pb 是个野值。

  • 可以先暂存原先的值，等到新值被构建好并没有出现问题时，再将旧值抛弃。

  • 可以使用条款 29 的 copy and swap 技术

    • ```c++
      class Widget
      {
      public:
      void swap(Widget &rhs); // 交换 *this 和 rhs 的数据

      Widget &operator=(const Widget &rhs)
      {
      Widget temp(rhs); // 拷贝构造 rhs 的副本
      swap(temp); // 交换 *this 和 rhs 的副本
      return *this;
      }
      private:
      BitMap *pb;
      };

    • 优点是比较方便

条款 19：设计 class 犹如设计 type

在设计class之前，设计者必须扪心自问：

  • 新type的对象应该如何被创建和销毁？
    • 这决定了构造函数、析构函数、内存分配和释放函数如何设计
  • 对象的初始化和赋值有何区别？
    • 决定了构造函数与赋值操作符的的行为，以及差异。尤其需要注意的是，“初始化”、“赋值”是完全不同的概念，它们调用了不同的函数。
  • 新type的对象如果被pass by value，意味着什么？
    • 记住，拷贝构造函数定义了一个新类型的传值（pass-by-value）如何实现。
  • 新type的合法值是什么？
    • 通常，对于一个类的数据成员来说，仅有某些值的组合是合法的。那些组合决定了你的类必须维持的不变量。这些不变量决定了你必须在成员函数内部进行错误检查，特别是你的构造函数，赋值运算符，以及 “setter” 函数。它可能也会影响你的函数抛出的异常，以及你的函数的异常规范（exception specification）（你用到它的可能性很小）。
  • 新type需要配合某个继承体系吗？
    • 如果当前type继承自某些class，那你必然受到了那些class设计的束缚，尤其是virtual或者non-virtual成员函数的影响。
    • 如果你允许别的类继承该type，那请谨慎考虑析构函数是否需要virtual。
  • 新type需要什么样的转换？
    • 如果你允许T1被隐式转换为T2，就必须在T1内写一个类型转换函数，或者为T2写一个non-explicit-one-argument(可被单一实参调用）的构造函数。
    • 如果你只允许explicit构造函数存在，就必须撰写显式类型转换函数，并且禁止定义类型转换操作符或者non-explicit-one-argument构造函数。
  • 新type需要哪些操作符和成员函数？
    • 该问题的回答决定了class中需要声明的函数，其中有一些是成员函数，有些则不应该是。（Effective C++ 24,25,47）
  • 新type应当明确禁止哪些函数？
    • 该问题的回答决定了class中哪些函数需要声明为没有定义的private成员函数。（Effective C++ 6）
  • 谁会试图使用新type的成员？
    • 该问题的回答决定了成员函数的可见性，并且也决定了friend。
  • type的一般化程度如何？
    • 如果一般化的程度足够高，或许我们不应该定义一个class，而是应该定义一个class template。
  • 是否真的需要一个新type？
    • 如果你只是想增加了一个derived class为原有的class添加功能，先试试看能不能用一些non-member函数或者templates解决。

条款 20：宁以 pass-by-reference-to-const 替换 pass-by-value

  • 前者通常比较高效，并可避免切割问题
    • 传值方式传递对象，会使用copy构造函数，对于一些体量庞大的对象来说，pass-by-value 是非常耗时的操作
    • 除了效率方面的考量，传引用也可以避免对象切割的问题。当一个 derived class 对象以 by value 的方式传递到函数中，而函数参数的类型是 base class 类，那么 base class 的 copy 构造函数会被调用，导致传入的对象变成了 base class ，绝不会是你想要的 derived class。
  • 该规则并不适用内置类型，以及STL的迭代器和函数对象。对它们而言，pass-by-value往往比较适当
  • 引用在 C++ 编译器的底层中，往往通过指针实现。

条款 21：必须返回对象时，别妄想返回其 reference

  • 绝不要返回指针或引用指向一个局部栈对象
  • 返回引用指向一个堆对象
  • 返回指针或引用指向一个局部静态对象而有可能同时需要多个这样的对象。

条款 22：将成员变量声明为 private

类的成员变量最好是 private, ，然后使用函数做读/写访问。有两大原因：

  • 使用函数控制可以让成员变量的处理有更精密的控制，比如，你可以通过设置 getter 和 setter 实现只读访问、不准访问、读写访问，甚至只写访问的成员变量。

  • 封装。将成员变量隐藏在函数接口的背后，可以为所有可能的实现提供弹性

    • 如果有一个成员变量是 public ，并且在后续代码中，直接读写这个成员变量。那么，一旦你需要实现当该变量被写时通知其他对象或者控制该变量的正确赋值范围，你会发现很多代码都需要变更！这是一个非常可怕的工作量。

条款 23：宁以 non-member non-friend 替换 member 函数

non-member non-friend 函数： 普通函数(不是成员函数，且不是友元函数）

面向对象守则要求数据应该尽可能地被封装，而不是将数据以及操作数据的那些函数被捆绑在一块。

non-member non-friend 函数比member函数好在：

  • 更高的封装性
  • 只有 class 的成员函数和其 friend 函数能够直接操控private成员。而增加一个member 函数就意味着增加一个可以窥见类内部秘密的“知情人”，其封装性就会减弱。
  • 较低的编译相依度
    • 成为类的 non-member non-friend 函数并不意味它不可以是另一个 class 的 member ，比如可以设立某些工具类的 member 函数。在 C++ 中，更加自然的做法就是让 class 和这些 non-member non-friend 函数存在于同一个 namespace 中。因为 namespace 可以跨文件，那么这些 non-member non-friend 函数就可以按分类放在不同的文件内，当客户需要时，再 #include 相应的头文件，以此降低编译依赖性。C++ 的标准程序库就是这样组织起来的。

条款 24：若所有参数皆需类型转换，请为此采用 non-member 函数

异常安全函数保证

异常安全函数提供以下三个保证之一：

  • 基本承诺

    • 如果异常被抛出，程序的任何事物仍然保持有效状态。没有任何对象和数据结构因此遭到破坏，所有对象均满足内部前后一致（例如所有的class约束条件仍然有效），但程序的现实状态不可预料。
      以实例为背景，我们可以确保在更换背景时如果抛出异常，对象会拥有原背景图案，或者对象会更改为某个默认背景图案，用户无法预期究竟会发生哪种情况。

  • 强烈保证

    • 如果运行期间存在异常被抛出，程序状态不改变。也就是说，运行成功则完全成功，失败则函数调用前的状态。（跟数据库中的事务一样）
      这比基本承诺所提供的性能要强，因为它确保了程序只有两种状态（调用成功，调用前），而基本承诺具备多个多态的可能（调用成功，调用前，以及任一合法状态）。

  • 不抛出异常保证（C++11引入noexcept修饰符）

    • 最高异常安全保证，确保程序在运行期间不会抛出任何异常，永远可以完成预期功能。作用于内置类型的所有操作都提供了不抛出异常保证。

    • 我们可以用noexcept或者throw()说明指定某个函数不会抛出异常，比如

异常安全函数必须提供上述三种保证之一，否则则不具备异常安全性。

不抛出异常自然是异常安全的最高追求，但很难做到。比如只要我们使用了动态内存，就有可能会接触bad_alloc异常。对于大部分函数而言，我们往往只追求基本保证和强烈保证之间。

  2. copy and swap可能实现强烈保证，但其并非对所有函数都可实现或具备实现意义。
  3. 异常安全具备全局性，其等于最弱的局部异常保证。

条款30：透彻了解inlining的里里外外

inline函数的本质

将对此函数的每一个调用都以函数本体替换之

好处与坏处

好处：

  • inline函数实质就是代码替换，将一次函数调用替换成直接执行函数体，所以如果inline函数体很小，“函数本体”比“函数调用”所产出的代码更小，那么会增加执行效率

坏处：

  • 如果inline函数体很大，就会导致代码膨胀，程序体积太大，可能内存放不下，增加换页中断的次数，最终导致程序执行速度下降。
  • 如果改变了inline函数的实现，所有用到该函数的源文件都需要重新编译，而修改一个非inline函数，只需要重新编译包含该函数实现的源文件和重新链接即可。

模板方法设计模式

  • 使用 non-virtual interface idiom (NVI idiom，非虚拟接口惯用法），这是Template Method design pattern（模板方法设计模式）的一种形式。用 public non-virtual member functions（公有非虚拟成员函数）包裹可访问权限较小的 virtual functions（虚拟函数。

策略模式

函数指针

  • 用 function pointer data members（函数指针数据成员）代替 virtual functions（虚拟函数），一种 Strategy design pattern（策略模式）的显而易见的形式。

（改进）function对象

  • 将函数指针变成function对象，这样就允许使用兼容于你所需要的东西的 any callable entity（任何可调用实体）。这也是 Strategy design pattern（策略模式）的一种形式。

（再次改进）将function对象变为另一个类的虚函数

  • 将继承体系中的虚函数替换为另外一个继承体系中的虚函数，这是 Strategy design pattern（策略模式）的习以为常的实现。

条款36：绝不重新定义继承而来的non-virtual函数

  • 绝不重新定义继承而来的non-virtual函数
    • 因为声明一个普通成员函数的目的是为了让派生类继承函数接口和强制性实现（条款34），如果一定要重新定义，那么该函数应该声明为虚函数！

条款37：绝不重新定义继承而来的默认实参

  • 绝对不要重新定义一个继承而来的默认实参，因为默认参数值都是静态绑定，而虚函数——你唯一应该重写的东西——却是动态绑定。

默认实参是静态绑定的！

我们能够继承的3种成员函数：虚函数、普通成员函数和静态成员函数，然而重新定义一个普通成员函数和静态成员函数永远是错误的（条款36），所以本条款的讨论局限于继承一个带有默认实参的虚函数。

3.2-3.5 C++对象的内存布局（重要）

发表于 2022-02-28 更新于 2022-03-11 分类于 深度探索C++对象模型

对象的内存布局

C++类的成员包括：

  • 普通/静态数据成员
  • 普通/静态函数成员
  • 基类成员
  • 为了实现虚拟化的某些技术（虚函数指针和虚基类指针）

其中静态数据成员，普通/静态函数成员是存储在一个类对象的内存空间之外的，所以一个类对象的内存空间里只会有基类成员、普通数据成员和为了实现虚拟化的某些技术。

下面依次讨论以下5种情况下的内存布局：

  • 没有继承，没有虚函数
  • 加入单继承
  • 加入虚函数
  • 加入多重继承
  • 加入虚继承

需要注意，由于内存对齐的要求，不同成员之间其实并不是一定连续排序的，可能会有内存碎片的存在，为了方便，以下情况，都忽略了内存碎片的存在。

没有继承，没有虚函数

没有继承，没有虚函数的情况下，对象的内存空间内只有普通数据成员，而C++标准规定，在同一个access section（public、protected、private）内的数据成员按照声明的顺序排列，而不同access section的先后不做要求，但通常实现是按声明的顺序。

综上所述，数据成员是按照声明的顺序排列

只有单继承

C++标准规定，继承不会修改父类子对象的内存布局， 例如父类由于内存对齐产生的额外空间在子类中不会被消除, 而是保持原样. 所以下面的代码中, 子类大小是24 bytes, 而不是16 bytes.

其原因是如果消除了这些额外空间, 将子类对象赋值给父类对象时就可能会在父类对象的额外空间位置赋值, 这改变了程序的语义, 显然是不合适的.

由于一个构造函数总是先执行父类的构造函数，然后执行子类的构造函数（析构函数按相反的方向，先子类析构，再父类析构），所以在只有单继承的情况，一个子类对象的内存布局是：先父类子对象，然后是子类的数据成员。

加入虚函数

虚函数带来的开销是：

  • 虚函数表

    • 构建一个虚函数表，存放着所有该类相关的虚函数地址和类的类型信息（用于支持RTTI）

  • 虚表指针（vptr）

    • 类的对象内存空间内加入一个虚表指针，指向该类的虚函数表（一个类只有一个虚函数表，所有对象共享这个虚函数表）
    • 一个单继承的类，只有一个虚表指针

  • 加强构造函数

    • 编译器自动在构造函数中插入代码，设定虚表指针的值。
    • 设定时机在基类的构造函数之后，初始化子类数据成员之前。
    • 考虑一个单继承的子类。构造子类对象时会先执行父类的构造函数，在父类的构造函数中，虚表指针的值会被先设定为指向父类的虚函数表，然后在子类的构造函数中，虚表指针再被设定为指向子类的虚函数表，其目的是为了假如父类的构造函数中，调用了一个虚函数，能够正确的执行父类实现的虚函数，而不是子类实现的虚函数。

  • 加强析构函数

    • 编译器自动在析构函数中插入代码， 析构函数需要能够重设虚表指针的值，在调用父类的析构函数之前，使其指向父类的虚函数表（析构，先析构子类，再析构父类，与构造函数相反）。

  • 通过指针或者引用调用虚函数的时候

    • a->func(); 会被编译器转化为 (*a->vptr[0])();, 即发生动态绑定，实际调用的函数需要等到执行期才能确定。

虚表指针存放的地址：

  • 可以放在对象内存空间的最后
    • 好处：对象的内存空间可以跟C兼容,可以将 C++对象直接传递给 c 函数进行处理。
  • 可以放在对象内存空间的开始
    • 好处：在多重继承中, 通过类成员指针访问虚函数会更容易实现, (其实就是不需要再调整this，加上vptr的偏移了)

加入多重继承

同样会先构造基类，然后再构造子类，虽然标准并没有规定不同基类在布局中的顺序, 但是大多数实现按照继承声明顺序安排. 假设多重继承有n条继承路线，那么子类就会有n个虚表指针（次要实例都有虚函数的前提下）。

而第二种就不会，增加虚基类，就增加虚表的表项，gcc的做法是将虚基类放在对象末尾,采用第二种方法。

3.0.2 内存对齐

发表于 2022-02-28 更新于 2022-03-11 分类于 深度探索C++对象模型

类的大小≥所有数据成员的大小总和，其原因是：

  1. 虚函数指针和虚基类指针的存在
  2. 内存对齐

内存对齐是什么？

  • 数据存储的内存地址，不能乱放，必须是某个值的整数倍。
  • 还有就是假如是一个类对象的话，它内部的数据成员，也不是简单的一个挨着一个这样存储的，会有一些空白位置存在（也就是内部碎片）

为什么需要内存对齐

  • 为了可移植性和提高处理器的性能。

  • 1）平台原因（移植原因）：不是所有的硬件平台都能访问任意地址上的任意数据的；某些硬件平台只能在某些地址处取某些特定类型的数据，否则抛出硬件异常。

    2）性能原因：数据结构（尤其是栈）应该尽可能地在自然边界上对齐。原因在于，为了访问未对齐的内存，处理器可能需要作两次内存访问；而对齐的内存访问仅需要一次访问。比如假如这个处理器只能访问偶数的内存，然后一次只能读4个字节，那么如果我一个int放在了奇数位置上，那么只能读两次，然后拼接起来。

detail

  • Linux GCC编译器环境下， 对于基本类型，起始位置要求是该类型大于与对齐系数较小值的整数倍。对于类类型， 除了起始位置有要求，每个数据成员相对于起始位置的偏移量也必须是类型大小与对齐系数较小值的整数倍。不是整数倍前面空出内存，直到偏移量是整数倍为止。同时考虑到为了让该类型的数组每个元素也符合对齐要求（起始位置），最后一个数据成员后面也可能会加一些空白
  • 可以用#pragma pack(2) 来指定对齐系数， Linux GCC 默认是4.
