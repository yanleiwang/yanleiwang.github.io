按值返回的函数是如何实现的？

编译器的经典做法是在函数参数里面新加一个引用的参数，然后通过拷贝构造函数来把返回值拷贝到这个参数里。

也就是一个双阶段转化：

  • 给该函数添加一个额外参数，类型是返回值类型的一个reference，用来放置被拷贝构造而得的返回值。
  • 在return指令之前，插入一个拷贝构造函数，将返回值当做这个新增参数的初值。然后函数返回void。

2.2 拷贝构造函数的构造操作

发表于 2022-02-22 更新于 2022-03-10 分类于 深度探索C++对象模型

什么情况会调用copy构造函数?

  • 用一个对象作为参数初始化另一个对象时.
  • 传值的方式调用一个函数时.
  • 传值的方式从函数返回一个值时. (不考虑返回值优化)

编译器合成的拷贝构造函数机制

如果不显式定义拷贝构造函数或者 A(const &A rhs) = default， 编译器会生成一个合成的拷贝构造函数，除了一种特殊情况：

  • 当类的数据类成员析构函数（拷贝构造函数）是删除的或不可访问的，这个类的copy构造函数是delete的。

合成的拷贝构造函数其整体思路是用 default memberwise initialization 实现的，具体点就是以一个个member的搞，对于内置类型的（int，指针，数组）是以bitwise copy（位逐次拷贝，就是一个个位复制）实现的，而对于类类型，则是递归的调用的memberwise initialization实现的。

一个Word对象的编译器合成拷贝构造函数会拷贝其内置类型的 _occurs，然后对String member执行递归式memberwise initialization。

类似于默认构造函数，C++亦把拷贝构造函数区分为trivial（无用）与non-trivial（有用）两种，只有non-trivial的实体才会被合成于程序之中。

trivial 拷贝构造函数，其实就是bitwise copy，把被拷贝对象的内存空间原样拷贝过来，并不调用copy构造函数, 可能的实现方式如利用memcpy等, 因此效率高, 复制出的对象和原对象完全相同.

什么时候是有用拷贝构造函数？

当合成拷贝构造函数不能表现为按位逐次拷贝语义的时候。

  1. 当类含有类对象成员（有一个即可，不用要求全部）, 且这个成员有有用的copy构造函数或者是显式定义的
  2. 当类继承自一个基类, 并且基类有有用的copy构造函数或者是显式定义的copy构造函数
  3. 当类含有虚函数时
  4. 当类有虚基类时.

对于1和2, 由于copy对象时, 要copy数据成员和基类, 既然它们提供了copy构造函数, 就可以认为需要在它们的copy构造函数中进行某些bitwise copy无法实现的操作, 因此不能采用bitwise copy.

对于3, 由于含有虚函数, 所以需要初始化对象的vptr, 而vptr的值不一定等于被拷贝对象的vptr值, 因为我们可以用子类对象初始化父类对象， 此时子类对象的vptr指向的子类的虚函数表，与父类的不同，所以bitwise不能满足需求.

对于4, 由于含有虚基类, 情况同上，可以用子类对象初始化父类对象，子类对象和父类对象的内存布局不同。

总结

一个编译器合成的copy构造函数，大部分情况下是bitwise copy，但是当我们有个类对象成员，它的copy构造函数不是bitwise copy的时候，（类对象成员包括基类），或者当类有虚函数，虚基类的时候，不能对虚函数指针和虚基类指针简单粗暴的bitwise copy，这时候会对那些刺头进行 不是bitwise copy， 具体做法是调用 类对象成员的 “有用” copy构造函数 对虚函数指针和虚基类指针进行显式设定。

2.1 默认构造函数的构造操作

发表于 2022-02-22 更新于 2022-03-10 分类于 深度探索C++对象模型

什么时候默认构造函数是delete的？

当一个类，没有定义构造函数的时候，我们知道编译器会给你合成一个默认构造函数，但当有以下情况的时候，编译器并不会为你合成一个，也就是定义为删除：

  • 类的某个数据类成员，它的析构函数是删除或者无法访问的（access level）
  • 类的某个数据类成员无法默认构造的时候
    • 它就是无法默认构造，比如有个其他的构造函数了，却没定义默认构造函数.
    • 它是个引用，那么它必须得显式初始化，然后又没给它类内初始值
    • 它是个const，且没有类内初始值, 但是没有显式的默认构造函数.（const 对象必须要初始化，但是一个有显式的默认构造函数的const 也是ok的， 比如 const string s1； 不会报错. 它可能会有一个编译器合成的默认构造函数，但是对于const 对象来说，不顶事）
  • 以上数据类成员包括基类

默认构造函数会干什么事？

编译器合成版本的默认构造函数会按以下顺序执行：

  1. 调用虚基类构造函数（假如有虚基类）
  2. 调用所有基类的构造函数，以基类的声明顺序为顺序（假如有基类）
  3. 设定虚表指针的值，指向相应虚函数表。（假如有虚函数）
  4. 调用所有数据成员的构造函数，以声明顺序为顺序（假如有类类型的数据成员）

补充：

  • 其中1、2、4，假如初始化列表显示提供了参数，会被传递过去，调用相应构造函数，否则调用默认构造函数。

  • 自己定义的所有构造函数，在执行函数体之前，也会被编译器执行1-4的内容, 也就是步骤5才是执行函数体。

  • 对于步骤4，内置类型的，比如整型、浮点型、字符型、指针、内置类型数组等都不会被编译器初始化，这些是程序员需要要干的事情。

  • 只有在1,2,3,4至少存在一个步骤的时候，编译器才会为你实际合成一个默认构造函数，否则是不会被合成的, 也就是所谓的trivial construction（无用的构造函数）。

  • 假如一个类没有默认构造函数，编译器也不给你合（也就是被定义为删除的），你又默认初始化了一个这个类的对象，那就gg，编译器报错。并且下面这段代码也会报错：

加入封装后的成本

在C语言中，“数据”及“处理数据的操作”是分开声明的，这种处理方式被称为procedural（面向过程）。

而C++中，数据和处理数据的函数是封装在一个类里，这种处理方式被称为面向对象。

而封装其实并未带来任何成本：

  • data
    所有的数据均存储于Object内，就如同struct中的情况一样。

  • member function
    成员函数虽然包含于class的声明之内，但却不会出现在object中。每一个non-inline成员函数都只会生成一个函数实体。

C++在内存布局以及存储时间上的额外负担主要由virtual引起：

  • 虚函数（virtual function）
    实现函数执行期绑定。
  • 虚基类（virtual base class）
    实现“多次出现在继承体系中的base class在派生类中只存在一个可共享的实体”。

一般而言，我们没有理由认为C++就一定会比C庞大且迟缓。

C++对象模型

在C++对象中，data member分为两种：

  1. static
  2. non-static

function member分为三种：

  1. static
  2. non-static
  3. virtual
  • 不考虑继承：
    • 非静态数据成员被配置于对象内，处于同一个access section的数据，排列顺序与声明顺序一致。（不同access section的数据，排序顺序，标准未定义）
    • static数据成员， static 和 非static函数成员被存储于对象之外，一个类只会有一份。
    • 虚函数则以虚函数表和虚表指针实现。每一个有虚函数的类，会有一个虚函数表，里面包含了这个类的所有虚函数地址以及这个类的类型信息，而每一个对象内都有一个虚表指针（vptr），指向这个虚函数表，vptr的设定与重置都由构造、析构、拷贝赋值运算符自动完成， 并且执行时间在自定义代码之前！！
  • 加入单一继承：
    • 会先执行基类的构造函数（此时同样会设置vptr，不过vptr指向的是基类的虚函数表），然后执行子类的构造函数（设置vptr指向子类的虚表）。
  • 加入多重继承：
    • 从左到右执行基类的构造函数，然后再执行子类的构造函数，假如有n条继承路线（n个基类），就会有n个vptr，指向n个不同的虚函数表，详见（TODO！！！）
  • 加入虚继承：
    • 有多种实现方法：
      • 对象内加入一个指针，指向虚基类对象
      • 在虚函数表里加一项，维护虚基类对象的位置。

由此可见，一个对象的内存占用由以下三部分组成：

  1. 所有non-static data member所占大小
  2. 为了支持virtual而产生的额外开销（vptr)
  3. 内存对齐所带来的额外占用

指针的类型

  • 指针其实存的就是一个地址，不管啥类型的指针大小都是一样，在32位，指针大小4个字节，64位：8个字节。
  • 指针的类型只是告诉编译器如何解释某个特定地址中的内存内容及其大小
  • 由此引申开来，不能用void * 操作对象（因为不知道指向的地址的内容和大小），针对指针的static_cast，其实是一种编译器指令，并没有改变一个指针的值，只是影响编译器如何去解读地址。

杂

  • 多态只存在于public 继承体系，因为一个基类指针不能指向继承方式为protected or private的对象。
