4.3 指向类成员的指针（数据成员和函数成员）

发表于 2022-03-01 更新于 2022-03-12 分类于 深度探索C++对象模型

指向普通数据成员

  • 形式：
    • 类型 类名::* 指针名 = &类名::成员名
    • 如int Bear::* ptr = &Bear::val
    • 得到的结果是该成员在类布局中的相对偏移再加1，比如ptr是指向Bear类里int类型的指针， ptr的值是成员变量val在Bear类里的偏移+1
  • 调用方式
    • 对象名.*指向成员的指针名或者指向对象的指针->*指向成员的指针名
  • 注意点：
    • &Bear::val 的值是成员变量val在Bear类里的偏移 + 1

指向静态数据成员

  • 形式：
    • 类型 *指针名 = &类名::成员名
    • int *ptr = &Bear::cnt
    • 不同于普通数据成员， 静态数据成员是所有该类的对象共有的，所以ptr是指向int类型的指针，ptr的值是Bear::cnt在内存中的地址
  • 注意点：
    • 指向静态数据成员的指针，其实就是一个普通的指针，而值是静态数据成员在内存中的地址。

指向非static函数成员

  • 形式：

    • 返回类型 (类名::*指针名)(形参类型列表) = &类名::成员函数名
    • void (Bear::*ptr)(string) = &Bear::speak
    • ptr是指向Bear类里 void (string)类型的函数的指针， ptr的指向的是Bear类里speak函数成员。（当speak是一个普通函数成员（非虚函数）的时候，ptr的值是speak函数的地址）

  • 使用方式：

    • ```c++
      Bear obj;
      Bear *p_obj = &obj;
      void (Bear::*ptr)(string) = &Bear::speak;

      (obj.*ptr)(“hello, world”);
      (p_obj->*ptr)(“hello, world”);

  • 实现原理：

    1. 当被指向的函数是一个普通成员函数的时候，其值是这个成员函数的地址。

    2. 早期C++，只考虑单一继承，且非虚继承的情况下，当成员函数指针指向虚函数的时候，其值是该函数在虚表中的偏移。（因此 (ptr->*pmf)() 会被编译器重写为

      (*ptr->vptr[(int)pmf])(ptr), 而基类的虚函数和子类重写的虚函数，在各自的虚表中，偏移是一样的，所以(ptr->*pmf)() 实际调用的是Point3d::z() 而不是Point::z()

    • 早期的C++，把1和2两种值都放在同一个指针里面，而区分他们的方法是，限制限制虚表长度为128， 所以情况2，最大值为127. 而程序空间起始地址必定大于127（）, 因此可以通过将指针值和127做”&”(按位与)运算来判断是偏移还是函数地址.， 即

    • 而现在为了支持多重继承，虚继承，实现方法已经不同了，早期的一种实现方法是，将成员函数指针指向一个结构体, 包含this指针偏移, 虚基类指针偏移等等. 但是会带来一个问题，对于那些普通成员函数，会带来额外负担。而现在的一种实现是单继承, 多继承, 虚继承就采用不同的方式来实现，减小负担。（但具体怎么实现的不明）

指向static函数成员

  • 形式：
    • 返回类型 （*指针名）(形参类型列表) = &类名::成员函数名
    • void (*ptr)(string) = &Bear::speak
    • ptr是 void (string)类型的函数的指针，ptr的值是speak函数的地址）
  • 注意点：
    • 指向静态函数成员的指针，其实就是一个普通的指向函数的指针，值是静态函数成员在内存中的地址。

4.0 名字修饰

发表于 2022-03-01 更新于 2022-03-12 分类于 深度探索C++对象模型

c++会对一些函数、变量的名字做修改，使每个名字独一无二，目的是为了实现：

  • 函数重载

  • 子类的数据成员隐藏同名的父类数据成员

会对哪些名字进行修改？

  • 普通函数，类成员函数（包括普通成员函数，static成员函数，虚函数）
    • 根据函数的参数列表（参数的个数，参数的类型）+ 函数名， 不包括返回类型
    • 这也是函数重载实现的机制，同名但是实际不同名（因为参数列表不一样）。
  • 类的数据成员
    • 根据类名 + 数据成员名

使用 extern “C” 会压抑名字修饰

c++支持三种类型的成员函数，分别为

  • static
  • nostatic
  • virtual。

每一种调用方式都不尽相同。

非静态成员函数 （Nonstatic Member Functions）

  • 保证nostatic member function至少必须和一般的nonmember function有相同的效率是C++的设计准则之一。

  • 事实上在c++中非静态成员函数（nostatic member function）与普通函数的调用也确实具有相同的效率。因为编译器内部会将成员函数等价转换为非成员函数，所以本质上非静态成员函数就如同一个普通函数, 如一个非静态成员函数：

    X float Point::X()

    就相当于一个普通函数：

    float X(Point * const this)

    编译器具体是这样做的:

    • 改写成员函数的签名，使得其可以接受一个额外参数，这个额外参数即是this指针：

    • ```c++
      float Point::X();
      //成员函数X被插入额外参数this
      float Point:: X(Point* const this );
      // 当然如果成员函数是const的，插入的参数类型将为 const Point * const 类型。（指向常量的常量指针）

也就是通过虚函数指针找到虚函数表，然后根据固定的表项找到实际要执行的虚函数。

  • 其中vptr为指向虚函数表的指针，它由编译器产生。vptr也要进行名字修饰，因为一个继承体系可能有多个vptr。
  • 1是虚函数在虚函数表中的索引，通过它关联到虚函数function().

何时发生这种转换？

  • 当通过指针或者引用调用的时候，因为要调用的函数实体无法在编译期决定，必需待到执行期才能获得，
  • 但是当我们通过对象（不是引用，也不是指针）来调用的时候，就不会发生，因为在编译器要调用的函数实体已经被决定。此时调用的就是根据指针的静态类型决定的函数。

3.2-3.5 C++对象的内存布局（重要）

发表于 2022-02-28 更新于 2022-03-11 分类于 深度探索C++对象模型

对象的内存布局

C++类的成员包括：

  • 普通/静态数据成员
  • 普通/静态函数成员
  • 基类成员
  • 为了实现虚拟化的某些技术（虚函数指针和虚基类指针）

其中静态数据成员，普通/静态函数成员是存储在一个类对象的内存空间之外的，所以一个类对象的内存空间里只会有基类成员、普通数据成员和为了实现虚拟化的某些技术。

下面依次讨论以下5种情况下的内存布局：

  • 没有继承，没有虚函数
  • 加入单继承
  • 加入虚函数
  • 加入多重继承
  • 加入虚继承

需要注意，由于内存对齐的要求，不同成员之间其实并不是一定连续排序的，可能会有内存碎片的存在，为了方便，以下情况，都忽略了内存碎片的存在。

没有继承，没有虚函数

没有继承，没有虚函数的情况下，对象的内存空间内只有普通数据成员，而C++标准规定，在同一个access section（public、protected、private）内的数据成员按照声明的顺序排列，而不同access section的先后不做要求，但通常实现是按声明的顺序。

综上所述，数据成员是按照声明的顺序排列

只有单继承

C++标准规定，继承不会修改父类子对象的内存布局， 例如父类由于内存对齐产生的额外空间在子类中不会被消除, 而是保持原样. 所以下面的代码中, 子类大小是24 bytes, 而不是16 bytes.

其原因是如果消除了这些额外空间, 将子类对象赋值给父类对象时就可能会在父类对象的额外空间位置赋值, 这改变了程序的语义, 显然是不合适的.

由于一个构造函数总是先执行父类的构造函数，然后执行子类的构造函数（析构函数按相反的方向，先子类析构，再父类析构），所以在只有单继承的情况，一个子类对象的内存布局是：先父类子对象，然后是子类的数据成员。

加入虚函数

虚函数带来的开销是：

  • 虚函数表

    • 构建一个虚函数表，存放着所有该类相关的虚函数地址和类的类型信息（用于支持RTTI）

  • 虚表指针（vptr）

    • 类的对象内存空间内加入一个虚表指针，指向该类的虚函数表（一个类只有一个虚函数表，所有对象共享这个虚函数表）
    • 一个单继承的类，只有一个虚表指针

  • 加强构造函数

    • 编译器自动在构造函数中插入代码，设定虚表指针的值。
    • 设定时机在基类的构造函数之后，初始化子类数据成员之前。
    • 考虑一个单继承的子类。构造子类对象时会先执行父类的构造函数，在父类的构造函数中，虚表指针的值会被先设定为指向父类的虚函数表，然后在子类的构造函数中，虚表指针再被设定为指向子类的虚函数表，其目的是为了假如父类的构造函数中，调用了一个虚函数，能够正确的执行父类实现的虚函数，而不是子类实现的虚函数。

  • 加强析构函数

    • 编译器自动在析构函数中插入代码， 析构函数需要能够重设虚表指针的值，在调用父类的析构函数之前，使其指向父类的虚函数表（析构，先析构子类，再析构父类，与构造函数相反）。

  • 通过指针或者引用调用虚函数的时候

    • a->func(); 会被编译器转化为 (*a->vptr[0])();, 即发生动态绑定，实际调用的函数需要等到执行期才能确定。

虚表指针存放的地址：

  • 可以放在对象内存空间的最后
    • 好处：对象的内存空间可以跟C兼容,可以将 C++对象直接传递给 c 函数进行处理。
  • 可以放在对象内存空间的开始
    • 好处：在多重继承中, 通过类成员指针访问虚函数会更容易实现, (其实就是不需要再调整this，加上vptr的偏移了)

加入多重继承

同样会先构造基类，然后再构造子类，虽然标准并没有规定不同基类在布局中的顺序, 但是大多数实现按照继承声明顺序安排. 假设多重继承有n条继承路线，那么子类就会有n个虚表指针（次要实例都有虚函数的前提下）。

实现虚继承的一大难点在于需要找到一个足够有效的方法，如何实现只有一个虚拟基类？

主流方案是分成两个部分，一个不变区域，一个共享区域，不变区域放的是非虚基类部分，而把虚基类作为共享部分。而虚基类的直接派生类如何存取虚基类，有两种方法：

  • 在每个直接派生类里面放一个指针指向这个虚拟基类
  • 在虚表中添加一项, 记录基类对象在对象中的偏移, 从而获得其地址.

第一种实现模型主要有两个缺点：

  1. 每一个virtual base class都存在一个与之对应的指针，但我们希望空间开销固定，不能因为virtual base class数目的增长而增长。
  2. 继承体系发生纵向增长后，间接存取的层次也将提高，我们希望时间开销固定，不随着继承体系扩张而增长。

而第二种就不会，增加虚基类，就增加虚表的表项，gcc的做法是将虚基类放在对象末尾,采用第二种方法。

  • 非静态数据成员

    • 通过对象名（非引用、非指针类型）访问非静态数据成员时（包括虚基类的数据成员），与访问静态数据成员也没有区别，因为存取的做法是对象的起始位置+一个偏移量，偏移量固定好了，在编译的时期就已经知道了，所以这个操作是在编译期做的。

      • 访问虚基类数据成员时，需要在运行时才能确定, 因为无法确定指针所指对象的实际类型, 也就不能判断对象的内存布局, 也就不知道对象中该数据成员的偏移

3.0.2 内存对齐

发表于 2022-02-28 更新于 2022-03-11 分类于 深度探索C++对象模型

类的大小≥所有数据成员的大小总和，其原因是：

  1. 虚函数指针和虚基类指针的存在
  2. 内存对齐

内存对齐是什么？

  • 数据存储的内存地址，不能乱放，必须是某个值的整数倍。
  • 还有就是假如是一个类对象的话，它内部的数据成员，也不是简单的一个挨着一个这样存储的，会有一些空白位置存在（也就是内部碎片）

为什么需要内存对齐

  • 为了可移植性和提高处理器的性能。

  • 1）平台原因（移植原因）：不是所有的硬件平台都能访问任意地址上的任意数据的；某些硬件平台只能在某些地址处取某些特定类型的数据，否则抛出硬件异常。

    2）性能原因：数据结构（尤其是栈）应该尽可能地在自然边界上对齐。原因在于，为了访问未对齐的内存，处理器可能需要作两次内存访问；而对齐的内存访问仅需要一次访问。比如假如这个处理器只能访问偶数的内存，然后一次只能读4个字节，那么如果我一个int放在了奇数位置上，那么只能读两次，然后拼接起来。

detail

  • Linux GCC编译器环境下， 对于基本类型，起始位置要求是该类型大于与对齐系数较小值的整数倍。对于类类型， 除了起始位置有要求，每个数据成员相对于起始位置的偏移量也必须是类型大小与对齐系数较小值的整数倍。不是整数倍前面空出内存，直到偏移量是整数倍为止。同时考虑到为了让该类型的数组每个元素也符合对齐要求（起始位置），最后一个数据成员后面也可能会加一些空白
  • 可以用#pragma pack(2) 来指定对齐系数， Linux GCC 默认是4.

2.4 构造函数初始化列表

发表于 2022-02-23 更新于 2022-03-10 分类于 深度探索C++对象模型

构造函数的初始化列表的目的：一是可以提升效率，还有一个是有时候必须要这么做。

  • 提升效率：

因为编译器会对构造函数进行扩充，先对数据成员进行初始化（内置类型不初始化，类类型调用默认构造函数），再执行构造函数体。

那么如果在函数体内对数据赋值，就会存在先进行初始化，然后进行赋值操作（可能还会有先构造一个临时对象，然后进行赋值操作，再析构这样的操作）

而如果用初始化列表，就能初始化一步到位（实际编译器转换后的代码是先执行初始化列表（对应的构造函数），再执行函数体的代码）

编译器生成的代码，初始化顺序是类定义时数据成员的声明顺序决定，不是由列表排列顺序决定。（不过，数据成员是内置类型的话无所谓，你放在初始化列表里面，还是放在函数体里面，都ok，因为编译器不会对一个内置类型初始化作什么动作）。

  • 必须这么做
    • 有引用或者const对象，这种必须要初始化的成员的时候
    • 还有就是你要调用某个成员的非默认构造函数，或者这个成员它没有默认构造函数的时候（如果这样，会报错）
